If we define k to be *just* the gate complexity, and assume the number of witness polys t is constant, I think you can always get O(ndk). This retrieves the ndlogd, but is more general.
Say we are doing a sumcheck over Q(x)=P(f_1(x),..,f_t(x)) - i.e. P is the thing requiring k ops to compute and x here is a vector.
To compute the first univariate you need to compute Q at (n/2)d points - where x_2,..,x_m range over the hypercube and x_1 ranges over some d values.


you set all variables except the first to fixed values b_2,..,b_m in the cube and are left with some Q*(x)=P(L_1(x),..,L_t(x)) where now x is single element, and the L_i are linear.
You can in time O(td)=O(d) evaluate all the L_i at d points a1,..,a_d.
Now at each such point a_i you can evaluate in k ops Q*(a)=Q(a,b_2,..,b_m).

I think in the end it’s the circuit complexity of symbolically evaluating n gate circuits. There are a lot of factors that play a role and it seems impossible to give one optimal algorithm. As Binyi and Ariel mentioned if I have a x^d Gate (within the larger gate circuit) I can first evaluate the input x (which is a polynomial) at sufficiently many points. For a linear x this is just straight line eval for larger it might make sense to use an FFT.
Another interesting case is something like \prod_t (w_i^d+w_{i+1}). Here it makes sense to first evaluate each (w_i^d*w_{i+1}) term and then use pairwise polynomial multiplication to compute the final product. This takes time O(d log d*t log^2 t *n ) vs O(d log d t^2 n) for the evaluate everything at d*t points algorithm and then combines.
