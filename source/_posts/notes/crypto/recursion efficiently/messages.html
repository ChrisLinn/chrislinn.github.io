Daira Hopwood
The recent paper https://eprint.iacr.org/2020/1618 makes it more feasible to do recursion via private aggregation without a cycle of curves, because the aggregation verifier only requires a constant number of operations in the "wrong" field
19:33
This approach requires that you carry a large state between proofs, though
19:34
(the current state of the computation is still verifiable with a succinct proof, but that proof isn't sufficient on its own to make subsequent proof steps)
19:44
However, I think this would in practice be much more efficient than any approach using FRI and/or GKR.
19:48
Note that in PLONK-style arithmetization it's possible to do multiple lookups per row (we have support for this in the Halo 2 prototype), so that can further reduce the overhead of wrong-field arithmetic even for circuits where you can't afford very large tables. This comes at the cost of an increase in proof size (linear in the number of lookup arguments)
19:53
Btw, if there's a way to optimize wrong-field arithmetic based on the field sizes being close or having a particular structured difference, I couldn't find it. The bottleneck is in reduction, specifically decomposing the unreduced product. The techniques that are usually used outside circuits don't help because they are based on decomposition being a cheap operation, and in circuits it isn't.
19:58
(we have a lot of control over the difference between the base and scalar field sizes if we use CM curves, so it's conceivable that helps, but as I say, I couldn't find anything)
